# Roadmap

This roadmap separates engineering maturity from investment-performance proof.

## Already Represented In This Snapshot

- local app server
- HTS-style dashboard
- MCP server surface
- AI staff workflow concepts
- candidate/radar/replay/review structures
- Research Forge integration
- external engine worker interfaces
- paper/live safety separation
- reconciliation-focused trading operation structures
- tests covering many safety and workflow contracts
- independent internal-developer sidecar
- heartbeat/progress-aware watchdog handoff
- incident, report, advice, event, and verified-playbook ledgers
- Telegram and launcher operational-health visibility
- GPT/MCP external-guidance intake with quarantine and revalidation
- local Ollama startup recovery with bounded CPU fallback

## Near-Term Public Hardening

1. Add synthetic demo data for first-run onboarding.
2. Add a one-command paper-mode smoke test.
3. Add screenshots after confirming no private data is visible.
4. Split optional heavy engine dependencies into extras.
5. Create a compact public MCP facade with 18-20 read-only tools.
6. Add GitHub Actions for syntax checks and a lightweight test subset.
7. Add an English quick-start video or short walkthrough.
8. Add a sanitized internal-developer demo trace to first-run onboarding.

## Research and Evidence Hardening

1. Complete point-in-time universe evidence.
2. Improve official corporate-action event history coverage.
3. Add more blind and out-of-sample validation flows.
4. Add Monte Carlo stress reports to the public evidence model.
5. Maintain strategy A/B freeze records for forward paper comparison.
6. Separate research-only results from live-trading claims.

## Trading Operations Hardening

1. Keep market-hours focus on monitoring and candidate execution readiness.
2. Continue order/fill/balance/PnL reconciliation improvements.
3. Improve simple Telegram reporting for live/paper distinction.
4. Record every buy/sell reason and exit plan in a queryable journal.
5. Strengthen blocked-order explanations and recovery actions.

## Operations and Recovery Hardening

1. Accumulate long-duration evidence for heartbeat and watchdog stability.
2. Test additional real-world failure modes without relaxing the action allowlist.
3. Add user-facing acknowledgement and closure controls for recovered incidents.
4. Add token/advice retention controls for public MCP deployments.
5. Keep code changes, credential changes, security changes, and live-order actions human-reviewed.

## Packaging

1. Produce a clean desktop launcher package.
2. Keep private runtime folders outside the source checkout.
3. Add an installer-time security checklist.
4. Provide separate install profiles:
   - research-only
   - paper trading
   - local live-trading pilot
   - public MCP review

## Performance Proof

CodexStock should not claim investment superiority from architecture alone.

Evidence still needed:

- at least 90 trading days of forward paper/live observation
- stable operation logs
- realistic slippage and cost calibration
- failure-mode records
- benchmark comparisons
- independent review of assumptions

## Public Philosophy

CodexStock should be evaluated as:

- a personal AI investment operations platform
- a research and safety architecture
- an auditable workflow system

It should not be marketed as:

- a guaranteed-profit system
- a black-box trading signal seller
- a replacement for personal responsibility
